# Threat Level Access (TLA) Confirmation System

## Overview

Every tool call is classified by the host at dispatch time
(`jarvis/core/threat_level.py`); calls rated ELEVATED or above are gated
behind user approval **before** the tool runs.  Tool metadata
(`threat_level`, legacy `confirmation_required`) can raise but never lower
the host-computed level.  The confirmation is handled entirely by the
confirmation manager — the LLM is never in the confirmation loop, which
prevents it from misrepresenting the action.

## Design Principles

1. **The host decides; tool authors can only raise** — every call's effective
   threat level is `max(host floor, manifest declaration, payload scan)`
   (`jarvis/core/threat_level.py`).  The host classifies command/script
   execution tools as DANGEROUS regardless of the manifest, so a tool author
   can raise a tool's level but never lower it below the host floor.
2. **LLM is out of the loop** — confirmation prompts are generated by the
   kernel from tool metadata, not by the LLM.  This prevents the LLM from
   saying "I'm checking the weather" while actually deleting files.
3. **Non-blocking / event-driven** — confirmation never blocks the event loop.
   The manager sends a notification and returns immediately.  The response
   arrives later as a `CONFIRMATION_RESPONSE` event through the EventMerger,
   keeping the LLM and dispatcher fully async.
4. **Channel-agnostic** — confirmation works across desktop notifications,
   CLI stdin, and socket IPC.  The kernel picks the best available channel.
5. **Persistent by default** — pending confirmations are tracked in a
   queryable list (`ConfirmationManager.list_pending()`, exposed via CLI
   and socket), reviewable at any time rather than expiring on a clock.
   Auto-deny-on-timeout is opt-in (`CONFIRMATION_TIMEOUT` > 0) for
   unattended/headless setups that want the old fail-safe behavior back.

## Configuration

| Variable | Default | Description |
|---|---|---|
| `CONFIRMATION_MODE` | `smart` | `allow_all` / `smart` / `ask_all` |
| `NOTIFICATION_SILENT` | `false` | Suppress desktop notifications (socket/CLI only) |
| `CONFIRMATION_TIMEOUT` | `0` | Seconds before auto-deny; `0` disables it (persistent list instead) |

### Modes

- **`allow_all`** — Never ask.  For power users and trusted environments.
- **`smart`** — Confirm any tool whose effective threat level is ELEVATED or
  above: `max(host floor, manifest declaration, payload scan)`.
- **`ask_all`** — Ask for every tool call, regardless of metadata.

## MCP Tool Metadata

MCP server developers add these fields to their tool definitions:

```json
{
  "name": "filesystem.delete",
  "description": "Delete files or directories",
  "inputSchema": { ... },
  "confirmation_required": true,
  "notification_silent": false
}
```

| Field | Type | Default | Description |
|---|---|---|---|
| `threat_level` | `string` | `"safe"` | One of `safe` / `elevated` / `dangerous` / `forbidden`; tools at `elevated` or above are confirmed in smart mode |
| `confirmation_required` | `bool` | `false` | **Legacy** — equivalent to `threat_level: "elevated"`; can only raise, never lower, the host-computed level |
| `notification_silent` | `bool` | `false` | Suppress desktop notification for this tool |

## Async Architecture

The confirmation system is fully non-blocking, matching JARVIS's event-driven
architecture.  This is critical because JARVIS processes reminders, signals,
and user input concurrently — a blocking confirmation would freeze everything.

### Flow

```
User: "delete my temp files"
    │
    ▼
  ROOT → action=dispatch, tool=filesystem.delete
    │
    ▼
  _dispatch_send() — checks tool metadata
    │
    ├─ No confirmation needed → dispatch immediately, return result
    │
    └─ Confirmation needed:
        │
        ├─ 1. Stash tasks in ConfirmationManager._pending
        ├─ 2. Send notification (fire-and-forget):
        │       ├─ Desktop: notify-send runs as async subprocess
        │       ├─ Socket:  JSON broadcast to connected clients
        │       └─ CLI:     input() in executor thread
        ├─ 3. Start timeout task, only if CONFIRMATION_TIMEOUT > 0
        │      (default 0 -- stays pending indefinitely, reviewable via
        │      list_pending()/CLI/socket instead of expiring)
        └─ 4. Return {"awaiting_confirmation": True} immediately
            │
            ▼
        EVENT LOOP CONTINUES FREELY
        (reminders, signals, new user input all flow normally)
            │
            ▼
        User clicks Allow/Deny (or timeout fires)
            │
            ▼
        Response injected into EventMerger as CONFIRMATION_RESPONSE event
            │
            ▼
        _on_confirmation_response():
            │
            ├─ Approved → dispatch approved tasks, feed result to ROOT
            └─ Denied   → feed USER_DENIAL to ROOT
                │
                ▼
            LLM continues naturally:
            "Alright, I won't delete those files."
```

### Why non-blocking matters

Without async confirmation, this would happen:

```
t=0s   User: "delete temp files"     ← confirmation blocks here
t=0s   Timer: REMIND goal X          ← STUCK, can't process
t=5s   User: "what time is it?"      ← STUCK, can't process
t=30s  Confirmation times out         ← finally unblocks
t=30s  Now processes stale events     ← bad UX
```

With async confirmation:

```
t=0s   User: "delete temp files"     ← notification sent, returns immediately
t=0s   Timer: REMIND goal X          ← processed normally ✓
t=5s   User: "what time is it?"      ← processed normally ✓
t=8s   User clicks Allow             ← CONFIRMATION_RESPONSE event
t=8s   Tasks dispatched              ← resumes exactly where it left off
```

## Confirmation Channels

Channels are tried in priority order.  All channels are non-blocking — they
send the notification and inject the response back via EventMerger.

### 0. TUI Modal (highest priority)

When the Textual TUI is running, confirmations render as a `ConfirmModal`
(`push_screen_wait`, wired in `tui/lifecycle.py`) and the answer is injected
as a `confirmation_response` event.

### 1. Desktop Notification

Uses `notify-send` (Linux / libnotify) with Allow / Deny action buttons.
Runs as an async subprocess — when the user clicks, the response is injected
into the EventMerger.  The notification body is the rendered command line
(#186), and a batch produces one Allow/Deny notification per task
(notify-send supports only two actions), aggregated into a per-task decision.
Dismissing or letting a notification expire leaves the request pending.

```
┌─────────────────────────────────────────┐
│  JARVIS — Confirmation Required         │
│                                         │
│  shell.run_command: rm -rf /tmp/cache   │
│                                         │
│  [Allow]         [Deny]                 │
└─────────────────────────────────────────┘
```

Skipped when `notification_silent` is `true` (per-tool or global).

### 2. Socket IPC

Sends a structured JSON message on the output socket:

```json
{
  "type": "confirmation_request",
  "id": "abc123",
  "tools": ["com.example.fs.delete"],
  "details": [{"index": 0, "tool": "com.example.fs.delete", "params": {"path": "/tmp/*"}, "command": "fs.delete: /tmp/*"}],
  "timeout": 30
}
```

External apps (Flutter widget, Electron tray, web dashboard) render their
own confirmation UI and respond on the input socket with either shape:

```json
{"type": "confirmation_response", "id": "abc123", "approved": true}
{"type": "confirmation_response", "id": "abc123", "approved_indices": [0, 2]}
```

`approved` is all-or-nothing; `approved_indices` approves per task (indices
into `details`; unlisted indices are denied — see Denial Flow).

This is the primary channel when `notification_silent` is `true` — it lets
apps fully customize the confirmation experience.

### 3. CLI / TTY Fallback

Runs `input()` in an executor thread (non-blocking to the event loop).
Response injected via EventMerger when the user types y/N.

```
[JARVIS] Confirmation required
  Tools: filesystem.delete
  Approve? [y/N]: _
```

## Denial Flow

When the user denies a tool:

```
CONFIRMATION_RESPONSE {approved: false}
    │
    ▼
  _on_confirmation_response():
    │
    ▼
  ROOT context:
  "USER_DENIAL: Action filesystem.delete was denied by the user"
    │
    ▼
  LLM continues conversation naturally:
  "Alright, I won't delete those files. Would you like me to
   do something else instead?"
```

The denial is sent as `USER_DENIAL` context (not `DISPATCH_ERROR`) so
the LLM understands the user is still engaged and should keep talking.

For partial denials (batch of tasks where some are denied, others approved):

- Approved tasks execute normally.
- Denied tools are reported as `USER_DENIAL` alongside `DISPATCH_RESULT`.

## Event Types

The EventMerger supports five event types:

| Event | Source | Handler |
|---|---|---|
| `USER_INPUT` | Voice, CLI, socket | `_on_user_input()` |
| `DISPATCH_SIGNAL` | Dispatch binary | `_on_dispatch_signal()` |
| `DISPATCH_SIGNAL_BATCH` | Dispatch binary (merged `fire_wake=false` group, #189) | handled as one ROOT turn via the dispatch-signal handler |
| `CONFIRMATION_RESPONSE` | Notification channels | `_on_confirmation_response()` |
| `SHUTDOWN` | Signal handler | Stops event loop |
