# Tool-Level Action (TLA) Confirmation System

## Overview

MCP server tools can declare `confirmation_required: true` in their metadata.
When JARVIS encounters such a tool at dispatch time, it gates execution behind
user approval **before** the tool runs.  The confirmation is handled entirely
by the kernel / confirmation manager — the LLM is never in the confirmation
loop, which prevents it from misrepresenting the action.

## Design Principles

1. **Tool developers decide** — the person who wrote the tool knows whether
   it is destructive.  They set `confirmation_required` once; JARVIS enforces.
2. **LLM is out of the loop** — confirmation prompts are generated by the
   kernel from tool metadata, not by the LLM.  This prevents the LLM from
   saying "I'm checking the weather" while actually deleting files.
3. **Channel-agnostic** — confirmation works across desktop notifications,
   CLI stdin, and socket IPC.  The kernel picks the best available channel.
4. **Deny-safe** — timeout and "no channel available" both default to denial.

## Configuration

| Variable | Default | Description |
|---|---|---|
| `CONFIRMATION_MODE` | `smart` | `allow_all` / `smart` / `ask_all` |
| `NOTIFICATION_SILENT` | `false` | Suppress desktop notifications (socket/CLI only) |
| `CONFIRMATION_TIMEOUT` | `30` | Seconds to wait before auto-deny |

### Modes

- **`allow_all`** — Never ask.  For power users and trusted environments.
- **`smart`** — Only ask when the tool has `confirmation_required: true`.
- **`ask_all`** — Ask for every tool call, regardless of metadata.

## MCP Tool Metadata

MCP server developers add these fields to their tool definitions:

```json
{
  "name": "filesystem.delete",
  "description": "Delete files or directories",
  "inputSchema": { ... },
  "confirmation_required": true,
  "notification_silent": false
}
```

| Field | Type | Default | Description |
|---|---|---|---|
| `confirmation_required` | `bool` | `false` | Whether user approval is needed |
| `notification_silent` | `bool` | `false` | Suppress desktop notification for this tool |

## Confirmation Channels

Channels are tried in priority order:

### 1. Desktop Notification (default)

Uses `notify-send` (Linux / libnotify) with Allow / Deny action buttons.
The process blocks until the user clicks or the notification expires.

```
┌─────────────────────────────────┐
│  JARVIS — Confirmation Required │
│                                 │
│  Tool: filesystem.delete        │
│  Delete files in /tmp           │
│                                 │
│  [Allow]         [Deny]         │
└─────────────────────────────────┘
```

Skipped when `notification_silent` is `true` (per-tool or global).

### 2. Socket IPC

Sends a structured JSON message on the output socket:

```json
{
  "type": "confirmation_request",
  "id": "abc123",
  "tool": "filesystem.delete",
  "summary": "Delete files in /tmp",
  "params": {"path": "/tmp/*"},
  "timeout": 30
}
```

External apps (Flutter widget, Electron tray, web dashboard) render their
own confirmation UI and respond on the input socket:

```json
{"type": "confirmation_response", "id": "abc123", "approved": true}
```

This is the primary channel when `notification_silent` is `true` — it lets
apps fully customize the confirmation experience.

### 3. CLI / TTY Fallback

Standard stdin prompt when no desktop or socket is available:

```
[JARVIS] Confirmation required
  Tool:   filesystem.delete
  Action: Delete files in /tmp
  Approve? [y/N]: _
```

## Denial Flow

When the user denies a tool:

```
KERNEL: tool denied
    │
    ▼
  Context fed back to LLM:
  "USER_DENIAL: Action filesystem.delete was denied by the user"
    │
    ▼
  LLM continues conversation naturally:
  "Alright, I won't delete those files. Would you like me to
   do something else instead?"
```

The denial is sent as `USER_DENIAL` context (not `DISPATCH_ERROR`) so
the LLM understands the user is still engaged and should keep talking.

For partial denials (batch of tasks where some are denied, others approved):

- Approved tasks execute normally.
- Denied tools are reported as `USER_DENIAL` alongside `DISPATCH_RESULT`.

## Architecture

```
User: "delete my temp files"
    │
    ▼
  ROOT → action=dispatch, tool=filesystem.delete
    │
    ▼
  _dispatch_send() — confirmation gate
    │
    ▼
  ConfirmationManager.should_confirm(tool_meta)
    │
    ├─ CONFIRMATION_MODE=allow_all → skip, execute
    ├─ CONFIRMATION_MODE=ask_all  → always confirm
    └─ CONFIRMATION_MODE=smart    → check confirmation_required
        │
        ├─ false → execute
        └─ true  → request_confirmation()
            │
            ├─ Desktop notification (if not silent)
            ├─ Socket JSON (if clients connected)
            └─ CLI [y/N] (if TTY available)
            │
            ├─ Approved → execute
            ├─ Denied   → USER_DENIAL context → LLM keeps talking
            └─ Timeout  → deny (safe default)
```
